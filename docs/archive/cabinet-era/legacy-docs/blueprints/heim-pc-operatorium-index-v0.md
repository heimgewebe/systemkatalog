## Thesis, antithesis, synthesis

Thesis: The heim-pc already combines local execution, repo observation, task coordination, knowledge work and infrastructure control.

Antithesis: The same system is overgrown. Broad observation showed many non-default checkouts, dirty worktrees and stale branches. More autonomy without better perception would increase risk faster than usefulness.

Synthesis: Treat the heim-pc as an operatorium: a controlled local system for seeing, judging and only then acting. The first optimization target is better perception, not stronger hands.

Alternative axis: do not ask only what Grabowski can do now. Ask which local facts must be true before any agent is allowed to act.

## Evidence snapshot

Observed local signals from 2026-07-03:

- Grabowski runtime was healthy; deployment integrity and audit chain were valid; kill switch was not engaged.
- Fleet contained heim-pc, heimserver and heimberry.
- Steuerboard reported 113 evaluated repo/worktree contexts, 78 non-default branches and 7 detached heads.
- A broader exploratory `.git` marker inventory found 174 checkouts, 134 non-default checkouts, 31 dirty checkouts and 41 behind-remote checkouts.
- tmux, browser-worker and GUI-worker histories exist and show that the PC is already used as an agent-capable operating surface.
- rLens latest-complete bundles existed for several core repos, but the listed bundle set was from 2026-06-05 and is therefore context, not current truth.

Interpretation: the evidence supports an operatorium model, but does not prove action readiness for any individual repo.

## Organ model

### Grabowski

Role: bounded executor and local operator hand.

Use for: health checks, bounded shell and Git work, task execution, worker control, fleet reads and audited file edits.

Do not use as: autonomous merge authority or cleanup authority.

### Steuerboard

Role: repo-state monitor.

Use for: branch drift, operational profile and read-only context.

Do not use as: action approval or branch safety proof.

### Bureau

Role: coordination organ.

Use for: task registry, runtime truth, receipts and handoff semantics.

Do not use as: second GitHub or silent task mutator.

### Cabinet

Role: meaning and ecosystem memory.

Use for: system-level context, organ mapping, contradiction tracking and decision memory.

Do not use as: passive document dump or confidence machine.

### Lenskit/rLens

Role: repo intelligence organ.

Use for: bounded repo context and agent handoff packs.

Do not use as: live truth when bundles are stale.

### Vibe-Lab and Claim-Audit-Lab

Role: method and claim laboratories.

Use for: run comparison, claim discipline and evidence evaluation.

Do not use as: vanity benchmark shelf.

### Infra and device repos

Role: body and environment.

Use for: host, service, network and device contracts.

Do not use as: implicit trust zone.

### Personal-work repos

Role: reality anchor.

Use for: study, work, teaching, music therapy and actual personal utility.

Do not use as: permanent backlog graveyard.

## Resonance and contrast check

Interpretation A: the system is too fragmented and needs cleanup.

Interpretation B: the system is a productive sediment layer where old branches preserve decisions and experiments.

Resolution: classify first, then archive or clean. Deletion without lifecycle labels is premature.

Interpretation A: the next step should be a cleanup campaign.

Interpretation B: the next step should be an observability campaign.

Resolution: observability first. Cleanup follows only after owner, purpose and risk are explicit.

## Epistemic gaps

- Hardware and local model capacity are missing, needed for reliable local-agent planning.
- Full open PR and issue state is missing, needed for cross-repo prioritization.
- Service contracts are missing, needed to distinguish expected surfaces from unknown surfaces.
- Vault/private content was intentionally not read, so knowledge conclusions remain structural rather than semantic.
- Fresh rLens bundles are missing for current handoff truth.

## Risk and benefit classes

Benefits:

- lower repo coordination cost
- fewer premature merges
- better agent routing
- better recovery from interrupted work
- stronger local knowledge continuity

Risks:

- meta-system overgrowth
- false confidence from stale indexes
- accidental mutation of dirty worktrees
- security blind spots from unclassified runtime surfaces
- agent work becoming easier than deciding whether work matters

Rule: every new operatorium feature should name the friction it reduces and the evidence that proves the reduction.

## First operating doctrine

1. Sense before acting.
2. Prefer read-only observation unless target repo and lifecycle state are explicit.
3. Do not treat stale bundles as current truth.
4. Do not clean dirty checkouts without owner and purpose classification.
5. Do not merge without current-head review evidence and all findings considered.
6. Keep Cabinet as the meaning layer.
7. Keep Grabowski as bounded executor.
8. Use Vibe-Lab when a repeated method is being evaluated.
9. Use Bureau when coordination or receipts matter.
10. Use Steuerboard as early repo-state context, not as approval.
